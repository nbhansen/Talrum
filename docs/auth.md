# Auth (Phase 2: stubbed)

Phase 2 is single-parent, single-iPad. Adding a real sign-in screen now would
be friction without any payoff, so auth is **stubbed** — the client signs in
automatically on every boot as a fixed seeded user.

## What's in place today

- `src/lib/localAuth.ts` exports `LOCAL_PARENT_ID` — a fixed uuid
  (`00000000-0000-0000-0000-0000000000a1`) used as the `owner_id` on every
  seeded row.
- `supabase/seed.sql` (generated by `scripts/gen-seed.ts`) inserts a matching
  row in `auth.users` with that exact uuid, email `stub@talrum.local`, and
  password `stub-password-dev-only`.
- `src/lib/supabase.ts::ensureStubSession()` calls
  `signInWithPassword({ email, password })` with those credentials when no
  session exists. After this, `auth.uid()` inside Postgres equals
  `LOCAL_PARENT_ID`.
- Every RLS policy in `supabase/migrations/*_rls.sql` is written against
  `auth.uid()`. Because the stub user **is** the owner, the policies pass
  trivially during dev.

## Why this works for the Phase 3 flip

Phase 3 adds a real sign-up / sign-in UI and deletes the stub. The RLS
policies don't change. Every seeded row simply now belongs to whichever real
user signed up first, and further users get their own rows via normal
app flows.

## Where to look when flipping the switch

Everything that needs to change for real auth is one grep away:

```sh
grep -rn "LOCAL_PARENT_ID\|TODO(phase 3): auth\|stub-password-dev-only" src/ scripts/ supabase/
```

Expected hits:
- `src/lib/localAuth.ts` — delete the constant.
- `src/lib/supabase.ts` — replace `ensureStubSession` with real sign-in UI
  state. Drop the `TODO(phase 3): auth` comment block.
- `scripts/gen-seed.ts` — remove the `authUserRows()` block. Seed rows
  should either use the first real user's uuid (prompted at DB reset) or
  simply not be shipped for production builds.
- `supabase/seed.sql` — regenerate after the above edit
  (`npm run seed:gen`).

## Security notes

- The stub email and password are in the repo. This is intentional and
  safe because the stub user only exists on local dev DBs (Supabase CLI
  runs on 127.0.0.1).
- **Do not run `supabase db push` against a production server with the
  current seed.sql.** The stub rows plus the dev password would leak into
  prod auth.users. The Phase 3 PR removes the stub block before any push
  happens.
